# 第 2 章 · 为什么做（Why）

## 2.1 市场背景与现状问题

休闲益智品类在移动端和 H5 端持续增长,以 1010!、Blockudoku、俄罗斯方块等为代表的方块放置类游戏日活跃用户在 2023-2025 年维持在数千万量级。这类游戏具有"即开即玩、单局 3-10 分钟、无需联网"的特点,极其适合碎片化时间场景。然而,市面上大量同类产品的道具系统单一、连击反馈感不足、缺乏长线留存设计,玩家容易产生疲劳感。BlockUp Puzzle 希望在经典方块放置玩法基础上,通过丰富的道具策略和连击奖励机制,提供更有成就感的游戏体验。

## 2.2 核心痛点（一句话）

方块消除类游戏缺乏策略深度和正向反馈,玩家易陷入重复感。

## 2.3 业务目标

- 验证"方块放置 + 道具策略 + 连击奖励"组合玩法对用户留存的拉升效果
- 公开测试阶段达到日活 1000+ 并观察 7 日留存率

## 2.4 成功指标

| 指标名 | 指标类型 | 当前值（如有） | 目标值 | 测量方式 | 测量周期 |
|---|---|---|---|---|---|
| 日活跃用户 | 北极星 | 0 | 1000+ | 页面访问统计 | 日 |
| 次日留存率 | 留存 | - | ≥30% | 独立用户回访 | 日 |
| 平均单局时长 | 激活 | - | 3-8 分钟 | 会话时长统计 | 周 |
| 连击触发率 | 观察 | - | ≥40% 局含连击 | 事件埋点 | 周 |

# 第 3 章 · 做给谁（Who）

## 3.1 目标用户画像

### 主画像（必填，1 个）

- **身份 / 职业**：休闲益智游戏爱好者,年龄 16-45 岁,碎片时间充裕的上班族和学生
- **规模**：全球 H5 休闲游戏玩家,千万级潜在受众
- **典型场景**：通勤路上、午休间隙、睡前,打开浏览器/PWA 应用玩 1-3 局放松
- **当前替代方案**：1010!、Blockudoku、俄罗斯方块、Wood Block Puzzle 等同类游戏
- **当前替代方案的致命问题**：大多数同类游戏道具系统简单、缺乏连击奖励设计,中期玩法单调无变化

### 次要画像（选填）

- **身份**：超休闲游戏速通/高分玩家
- **典型场景**：追求高分排行,反复研究最优放置策略
- **痛点**：大多数同类游戏没有足够的策略深度来支撑高阶玩法

### 排除画像

- 重度竞技游戏玩家（本产品为单人休闲向,无对战/排位系统）
- 6 岁以下儿童（界面含虚拟货币概念,不适合幼龄用户）

## 3.2 核心使用场景（User Story）

- 场景 A · 作为休闲玩家,我想快速开始一局方块消除游戏,以便在碎片时间放松大脑
- 场景 B · 作为玩家,我想通过策略性放置方块触发连击,以便获得更高的分数和成就感
- 场景 C · 作为玩家,我想在陷入困境时使用道具（炸弹/重力/刷新）解围,以便延长单局寿命
- 场景 D · 作为玩家,我想查看和挑战自己的最高分记录,以便保持长期游玩动力
- 场景 E · 作为玩家,我想在游戏结束后使用复活功能继续游戏,以便不浪费已积累的分数

## 3.3 核心 Journey（主路径）

### Journey A · 单局游戏完整流程

- **触发**：玩家打开游戏页面,点击开始游戏
- **路径**：
  1. 玩家看到 10×10 棋盘和底部随机生成的 3 个待放置方块
  2. 玩家拖拽一个方块到棋盘上的合适位置
  3. 方块放置成功,占据对应格子
  4. 系统检测是否有行/列被完整填满
  5. 若有完整行/列,该行/列的方块被消除并得分,触发视觉特效
  6. 若连续消除触发连击,连击面板显示连击数和进度条
  7. 3 个方块全部放置后,系统再次随机生成 3 个新方块
  8. 若棋盘无法容纳任何待放置方块,游戏结束
- **终点**：玩家看到本局分数与最高分对比,可选择重新开始或使用复活
- **失败态 / 放弃态叙事**：最常见情况是棋盘被填满后无处放置当前方块,玩家体验到游戏结束界面,显示本局得分并提供复活选项

### Journey B · 道具使用流程

- **触发**：玩家在游戏中遇到困局,希望通过道具化解
- **路径**：
  1. 玩家点击底部道具栏中的某个道具图标（炸弹/重力/刷新）
  2. 道具激活,对棋盘产生相应效果
  3. 道具库存减一,金币消耗相应数量
  4. 玩家继续正常游戏
- **终点**：困局化解,玩家继续游戏并有机会获得更高分数
- **失败态**：金币不足或道具库存为 0,无法使用,玩家只能依靠放置策略

# 第 4 章 · 做什么（What）

## 4.1 方案总览

### 一句话定位

BlockUp Puzzle 是一款面向休闲玩家的 H5 方块放置消除游戏,通过策略放置、连击奖励和道具系统提供有深度的益智体验。

### 一段话描述

BlockUp Puzzle 是一款基于网格的方块放置益智游戏。玩家在 10×10 棋盘上策略性地放置系统随机生成的不同形状方块,当某一行或列被完整填满时该行/列会被消除并得分。游戏内置 37 种不同的方块形状（从单格到五格复合形状）,配合连击系统给予连续消除额外奖励。三种策略道具（炸弹清除、重力下落、刷新替换）为玩家提供解围手段。金币经济体系支撑道具购买和复活机制,使玩家在追求高分的同时需要权衡资源分配,增加策略维度。

## 4.3 核心功能详述

### 4.3.1 功能 · 方块放置与消除

**一句话**：玩家将随机生成的方块形状拖拽放置到棋盘上,填满整行或整列时自动消除得分。

**用户价值**：提供核心的益智策略体验,玩家需要规划空间布局,兼顾当前放置与未来空间预留。

**体验流程**：
1. 玩家看到棋盘下方的 3 个待放置方块
2. 选择一个方块,拖拽到棋盘上方
3. 拖拽过程中目标位置高亮提示是否可放置
4. 松手后方块落入棋盘对应格子
5. 系统自动检测并消除所有已填满的行和列
6. 消除的行/列触发视觉特效和得分动画
7. 3 个方块全部放完后,新一轮 3 个随机方块出现

**预期效果**：玩家通过策略放置方块不断消除得分,体验空间规划的满足感。

**典型反例（非目标）**：不是俄罗斯方块——方块不会自动下落,没有时间压力,玩家可以从容思考放置位置。

### 4.3.2 功能 · 连击系统

**一句话**：连续多次消除行/列时触发连击倍数,给予额外分数奖励。

**用户价值**：激励玩家进行更深层次的空间规划,追求连续消除而非单次消除,增加游戏深度和爽感。

**体验流程**：
1. 玩家放置方块触发第一次消除
2. 连击计数器启动,界面显示"Combo x1"
3. 玩家在随后放置中再次触发消除
4. 连击数递增,显示"Combo x2",进度条推进
5. 连击倍数应用到得分计算中
6. 若一段时间内未触发消除,连击中断归零
7. 高连击时触发特殊光效动画

**预期效果**：有经验的玩家能通过精准布局打出高连击,获得成倍的分数回报和视觉奖励。

### 4.3.3 功能 · 道具系统

**一句话**：三种功能各异的道具帮助玩家在困境中获得翻盘机会。

**用户价值**：降低挫败感,给玩家"第二次机会",同时引入资源管理的策略决策。

**体验流程**：
1. 玩家在游戏中积累金币
2. 遇到困局时点击道具栏的道具图标
3. 系统扣减道具库存和对应金币
4. 道具效果立即生效：
   - 炸弹：点击棋盘区域,清除该区域方块
   - 重力：所有悬空方块向下坠落,压实空间
   - 刷新：当前 3 个待放置方块替换为新随机方块
5. 玩家借助道具效果继续正常游戏

**预期效果**：玩家能在即将结束时通过道具逆转局面,单局寿命延长,分数天花板提高。

**典型反例（非目标）**：道具不是用于"自动消除"或"自动得分"——它们只是创造空间或替换手牌,最终仍需玩家自行决策放置位置。

### 4.3.4 功能 · 金币经济与复活

**一句话**：金币作为游戏内通用货币,支撑道具购买和游戏结束后的复活功能。

**用户价值**：给予玩家在单局和跨局之间进行资源管理的决策空间,提升长线参与感。

**体验流程**：
1. 玩家在游戏过程中通过消除和连击积累金币
2. 金币显示在界面顶部,实时更新
3. 使用道具时消耗对应金币（炸弹 120、刷新 150、重力 200）
4. 游戏结束时,若金币充足可选择复活继续游戏
5. 复活后棋盘保持当前状态,玩家获得新的放置方块

**预期效果**：玩家在"攒金币以备不时之需"和"即时使用道具获取更高分"之间做出策略权衡。

# 第 5 章 · 怎么做（叙事层 · 技术实现意图）

## 5.1 技术路线总览

BlockUp Puzzle 采用纯前端 H5 技术方案,基于 Scirra Construct 游戏引擎构建。整体为单页 Web 应用,无需后端服务器。游戏逻辑、资源加载、状态管理全部在浏览器端完成。玩家数据（最高分、道具库存、金币）通过浏览器本地存储持久化。游戏支持 PWA 标准,可安装到设备主屏幕并具备离线运行能力。资源托管在 GitHub Pages 上,通过 CDN 分发。

## 5.2 AI 与规则的职责分工

本产品无 AI 决策,所有业务逻辑由固定规则实现。

| 决策 | 由谁决定 | 为什么这么分工 |
|---|---|---|
| 方块形状随机生成 | 规则（随机算法） | 37 种预定义形状的等概率随机选取,无需 AI 参与 |
| 行列消除判定 | 规则（遍历检测） | 填满检测是确定性逻辑,无模糊空间 |
| 连击判定与计分 | 规则（计数器） | 连续消除检测与倍数计算是固定公式 |
| 游戏结束判定 | 规则（遍历判定） | 检测所有待放置方块能否放入棋盘是确定性计算 |
| 道具效果执行 | 规则（即时逻辑） | 炸弹/重力/刷新效果明确,无需智能决策 |

## 5.3 关键技术选择的理由

选择 Scirra Construct 作为开发引擎而非原生 Canvas/WebGL 开发,原因在于：Construct 提供完整的事件驱动式游戏开发环境,能够快速迭代 2D 益智游戏的逻辑和动画,同时输出标准 H5 代码,兼容各主流浏览器。相比 Unity WebGL 导出方案,Construct 生成的包体更小、加载更快,更适合超休闲游戏的"即开即玩"场景。放弃原生开发方案是因为团队规模小,Construct 的可视化开发效率更高。

选择 GitHub Pages 静态托管而非云服务器部署,原因在于：纯前端游戏无需后端计算,静态托管零成本零运维,通过 CDN 天然获得全球访问加速。

## 5.4 对外依赖

- Scirra Construct 运行时：游戏引擎核心,负责渲染、事件系统、物理模拟
- Azerion SDK：第三方广告/分发库,用于商业化接入（可选）
- GitHub Pages：静态资源托管与分发
- 浏览器 localStorage：本地数据持久化,无外部数据库依赖

# 第 6 章 · 不做什么（Why-not）

## 6.1 非目标清单

| 事项 | 为什么不做 | 类型 |
|---|---|---|
| 多人对战/排行榜联网 | 产品定位为单人休闲,联网增加复杂度且需要后端 | 产品能力边界 |
| 付费关卡/付费解锁内容 | 当前阶段验证玩法,不做收入 | 时段限制 |
| 自定义主题/皮肤 | 非核心玩法,不影响验证 | 时段限制 |
| 社交分享/好友系统 | 无后端能力,不支持社交 | 产品能力边界 |
| 6 岁以下儿童适配 | 含虚拟货币概念,不适合幼龄用户 | 客户准入边界 |

## 6.2 显式排除场景

- 不支持多人同屏或联网对战
- 不提供关卡制/闯关模式,仅有无限模式
- 不支持自定义方块形状或棋盘大小
- 不提供充值/真实货币购买虚拟金币
- 不生成任何 AI 内容或动态内容
- 不支持存档/读档（退出即结束当前局）
- 不提供教学关卡或新手引导动画

## 6.3 未来规划摘要

- 每日挑战模式：每日固定方块序列,玩家比拼同一局最高分 — 目标 V2.0
- 主题皮肤系统：方块和棋盘的视觉主题切换 — 目标 V2.0
- 全球排行榜：接入轻量后端,支持分数排名 — 目标 V2.5
- 成就系统：累计消除数/连击数等成就徽章 — 目标 V2.0

# 第 7 章 · 合规与风险声明

## 7.1 法规与标准依据

- 《网络游戏管理暂行办法》：如面向中国大陆运营需申请版号或豁免备案
- GDPR / 个人信息保护法：游戏不收集个人信息,localStorage 数据不离开设备
- 未成年人保护相关法规：游戏内含虚拟货币概念,需标注适龄提示
- PWA 标准（W3C）：确保渐进式 Web 应用合规实现
